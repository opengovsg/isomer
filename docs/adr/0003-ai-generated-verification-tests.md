# A PR's "Manual Verification Steps" are AI-translated into a committed, permanent Playwright test

The `pull_request_template.md` "Manual Verification Steps" section is a free-text checklist the PR author writes for the release deployer to tick off by hand. When a PR is marked `ready_for_review`, a new IsoBot job (`claude-code-action`, gated to OWNER/MEMBER like `isobot-pr-review.yml`) reads that checklist via an `/isobot-verify` skill, generates a convention-following e2e Playwright test, and **commits it to the PR branch**. The test is reviewed by a human like any other code, then runs through the **existing** `end-to-end-tests` job in `ci.yml`; the **existing** `post-pr-comment.yml` surfaces its run video as the artifact link. The prose checklist stays — it is the human-readable spec; the generated test is its executable form.

## Considered Options

- **Keep it manual (status quo)** — the release deployer runs each checklist by hand at release time. Rejected: the verification is ephemeral, repeated every release, and produces no durable coverage.
- **Hand-author the test as a convention** — make "write a Playwright test for the new behavior" the manual-verification step. Reliable and zero codegen risk, but pushes the full authoring cost onto every author and doesn't lower the barrier the way the team wanted.
- **Structured step DSL** — replace free-text with an interpretable selector/assertion format. Removes ambiguity but forces authors to learn a DSL and obligates us to maintain the interpreter.
- **AI generates fresh in CI every run, ephemeral** — regenerate and run on every push, never commit. Non-deterministic, unreviewable, and opens a false-green hole (a hallucinated no-op assertion passes silently). Rejected.
- **AI generates once, commits, human reviews, permanent (chosen)** — a one-time authoring assist that produces a deterministic, reviewable, durable regression test. After the commit it is an ordinary e2e spec flowing through infrastructure that already exists.

## Consequences

- **The false-green risk is real, and human review is the only safety net.** An AI test can assert nothing (`expect(page).toBeTruthy()`) and pass, defeating the entire point of verification. The committed-and-reviewed flow exists precisely so a human reads the assertions before the test is trusted. Reviewers must read generated assertions critically, not rubber-stamp them.
- **Generation quality is the whole ballgame.** The e2e suite has hard conventions (per-role `storageState`, page objects, `getSeedSiteId()` seeding, idempotency under a shared DB, "don't couple to Chakra classes" — see `tests/e2e/README.md`). The `/isobot-verify` skill must be fed the README, `fixtures/`, exemplar specs, and `CONTEXT.md` (domain language) or it will emit flaky, convention-violating tests.
- **Generated tests block merge from day one.** They land in the required `end-to-end-tests` job. An early over-strict or flaky generated assertion can block unrelated PRs; the mitigation is reviewer diligence, not a soft-launch lane.
- **Idempotency is diff-based.** The job skips generation if the PR diff already adds or edits any `tests/e2e/**/*.test.ts` — so the bot's own first commit makes later runs a no-op, and a human-authored test suppresses generation entirely. This protects human edits from being clobbered on re-trigger.
- **Fork PRs are not covered.** The workflow's gate restricts execution to OWNER/MEMBER authors, and `GITHUB_TOKEN` cannot push to a fork's branch anyway. Internal Graphite stacked PRs (same-repo branches) work; external-contributor PRs will not receive an auto-committed test.
- **The generated test runs inline via a shared reusable workflow — `GITHUB_TOKEN` only, no PAT.** The e2e harness was extracted from `ci.yml` into a reusable workflow (`e2e-tests.yml`, `workflow_call`) called by both `ci.yml` and `isobot-verify.yml`. The `isobot-verify` `generate` job pushes the test commit with `GITHUB_TOKEN`, then its `e2e` job calls the reusable workflow on the PR head branch (running the new test), and a `comment` job posts the video-artifact link directly. *We considered* having the push re-trigger `ci.yml` instead, but a push made with `GITHUB_TOKEN` does not trigger new workflow runs (GitHub's recursion guard), which would have required a long-lived PAT / GitHub App token. Running e2e inline avoids that secret entirely. Trade-off: the inline run tests the head branch tip, not the PR merge ref — it is a smoke test of the generated test, not a replacement for `ci.yml`'s merge-ref e2e gate, which still runs on later pushes.
- **The generation job executes branch-controlled content under a write token.** `claude-code-action` runs with Bash over the checked-out PR branch while a `contents:write` credential is present. This is acceptable *only* because the workflow gate restricts to OWNER/MEMBER authors (excluding forks). The token is the ephemeral, repo-scoped `GITHUB_TOKEN` — a smaller blast radius than a long-lived PAT: it cannot reach other repos and expires with the run. Weakening the gate turns this into arbitrary-code-execution with a write token. A workflow-level `concurrency` group (keyed by PR) prevents racing pushes.
- **No-op on empty checklists.** PRs with an empty or unchanged-from-template "Manual Verification Steps" section (chores, dependency bumps) produce no commit and no comment.
- **The suite accrues AI-authored tests over time.** Each PR's verification becomes permanent coverage; this is the intended payoff but will require occasional curation for overlap.
