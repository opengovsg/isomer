# Merge Category into Tag Category; drop the single-select guarantee

**Category** stops being a distinct concept. A Collection Item's category becomes an ordinary **Tag Category** group — auto-migrated from the legacy `category` string into a group labelled `"Category"`, appended to the end of the Collection Index's `tagCategories` array so existing filter display order is unchanged. Its selections move from a dedicated field into the shared `tagged` array alongside every other tag. This means editors can now select **multiple** options from the former category group — the "exactly one category" rule that existed since `category` was a plain string is gone, with no field-level replacement. We accepted this because building a per-group single/multi-select toggle just to preserve one field's old behaviour isn't worth the schema and UI complexity it would add everywhere else. It also avoids a second data migration later: if we ever want custom sort ordering that spans both Category and Tags together, that's far simpler with one unified model than with two.

## Considered Options

- **Two schemas, unified only where they overlap** — give `CategoriesSchema` its own `label`/`isRequired` fields (mirroring `TagCategorySchema`) but keep it a separate single-select concept. Rejected: still two schemas to maintain, and doesn't resolve the duplication the original request was about.
- **Merge shape, keep single-select via a flag** (e.g. `isSingleSelect` on `TagCategorySchema`) — one schema, but category groups still enforce exactly-one-selection. Rejected: keeps a distinction alive as a field instead of a schema, which is a smaller win than a true merge, and no one asked for single-select tag categories elsewhere.
- **Full merge, multi-select allowed (chosen)** — one schema, one storage model, no special-cased selection cardinality.

## Consequences

- **Category loses its uniqueness constraint with no replacement.** Any code or admin workflow relying on "an item has at most one category" must be updated or removed; there is no schema-level way to re-impose it later without reintroducing the distinction this ADR removes.
- **Category regresses from backend-enforced to UI-only enforcement.** The legacy `category` field is required at the schema level today, rejected by AJV/zod on any write path. Its replacement (`isRequired` on the "Category" tag group) only gets a Save-button guard in `MetadataEditorStateDrawer` — the same UI-only standard every other required Tag Category already lives with (`isRequired` has never been backend-enforced for any tag category). Direct API calls or other write paths are not blocked from omitting a category. Accepted as sufficient for now; revisit if this turns out to matter in practice.
- **Deleting an entire tag category becomes a real risk for "Category" too.** Today there's no way to delete the whole category taxonomy in one action — only individual `categoryOptions` one at a time. Once "Category" is an ordinary `tagCategories` group, an admin can delete the entire group (and therefore every item's category assignment) via the same "delete filter" flow used for any other tag category.
- **Rollout is staged across three PRs**: (1) schema unification, (2) migration script (legacy `category` string → `tagCategories`/`tagged`) + Studio UI consolidation, (3) remove the legacy `category` field once migration is confirmed complete on all sites. The legacy `category` field must keep working until PR3 lands.
